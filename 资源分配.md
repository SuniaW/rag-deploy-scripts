# AgentX 分布式架构 · 三服务器资源分配

> 目标：秒级体验
> 核心理念：**同区域就近存储、推理节点同机调用、监控独立部署**

---

## 架构总览

| 角色 | 服务器 | 配置 | 部署服务 |
|:---|:---|:---|:---|
| **A·主推理节点**<br>`124.223.22.34` | **4核** / 4G / 上海 | Ollama (LLM + Embedding)<br>AgentX Java 后端 | **4核 CPU 是秒级关键**：1.5B 模型在 4 核上推理可达每秒 50 Token 以上。App 和模型同机，消除网络 IO 延迟。 |
| **B·就近存储节点**<br>`110.40.177.238` | 2核 / 4G / **70G盘** / 上海 | Milvus (向量库)<br>Redis (缓存)<br>MinIO (对象存储) | **70G 大盘承载海量向量数据**。与节点 A 同城上海，RAG 检索延迟可控在 5ms 内，实现秒级知识召回。 |
| **C·流量网关节点**<br>`8.140.221.150` | 2核 / 4G / 北京 | Nginx (反向代理 + 前端)<br>Jaeger (全链路追踪)<br>Swagger 文档 | **运维独立**：流量入口与后端分离，不占用推理和存储的内存。北京节点作为统一入口，分发到上海集群。 |

## 为什么这个架构能达到"秒级体验"？

### 1. 4 核 CPU 集中给推理

Ollama 在没有 GPU 的服务器上全靠 CPU 算，而本架构把唯一的 **4 核机器** 全给了 Ollama。`qwen2.5:1.5b` 实测 4 核推理速度约 **50 Token/s**，从用户感知上就是"即问即答"。

### 2. 同城上海，内网低延迟

服务器 A 和 B 都在上海，即使不是同一账号/VPC，通过公网 IP 通信延迟也在 **5ms 以内**。RAG 检索走 B、AI 推理走 A，全链路耗时可控。

### 3. 内存压力均匀分布

| 服务器 | 服务 | 内存消耗 | 余量 |
|:---|:---|:---|:---|
| **A** (4G) | Ollama 3G + Java 512M | ~3.5G | 500M |
| **B** (4G) | Milvus 1.8G + Redis 256M | ~2.8G | 1.2G |
| **C** (4G) | Nginx 128M + Jaeger 256M | ~0.9G | 3.1G |
| 合计 | | ~7.2G | 充分利用 12G 总资源 |

每台都有余量，不会 OOM。


## 部署步骤

### 前置条件：所有服务器安装 Docker

```bash
# 上传并执行
chmod +x docker/install_docker.sh && ./docker/install_docker.sh
# 然后重新登录终端使 docker 组权限生效
```

### 服务器 B — 存储层（先部署，因为 A 和 C 依赖它）

```bash
# 上传 server-B/ 目录到 110.40.177.238
scp -r server-B root@110.40.177.238:/usr/service/

# SSH 登录并执行
ssh root@110.40.177.238
cd /usr/service/server-B
chmod +x deploy.sh
./deploy.sh
```

### 服务器 A — 推理层

```bash
# 上传 server-A/ 目录到 124.223.22.34
scp -r server-A root@124.223.22.34:/usr/service/

# 注意：Java 代码需放在 /usr/service/spring-ai-rag/ 下（docker-compose build context）
# 上传 Java 项目后执行
ssh root@124.223.22.34
cd /usr/service/server-A
chmod +x deploy.sh
./deploy.sh
```

### 服务器 C — 网关层

```bash
# 上传 server-C/ 目录到 8.140.221.150
scp -r server-C root@8.140.221.150:/usr/service/

# 上传前端构建产物
scp -r dist/* root@8.140.221.150:/usr/service/server-C/data/web-ui/

# SSH 登录并执行
ssh root@8.140.221.150
cd /usr/service/server-C
chmod +x deploy.sh
./deploy.sh
```

## 安全建议

1. **所有服务器打开 Swap**
   ```bash
   fallocate -l 4G /swapfile
   chmod 600 /swapfile
   mkswap /swapfile
   swapon /swapfile
   echo '/swapfile none swap sw 0 0' >> /etc/fstab
   ```

2. **Redis 密码**：上线后修改 `redis.conf` 中的 `requirepass`

3. **Server C 使用 HTTPS**：用 Certbot 申请免费证书
   ```bash
   apt install certbot python3-certbot-nginx
   certbot --nginx -d your-domain.com
   ```

## 架构师总结

本方案通过 **同城双节点核心**（上海两台机器承载全部业务）+ **运维独立网关**（北京节点承载入口与监控）的部署，在 **12G 总内存、8 核 CPU** 的资源约束下，实现了秒级推理体验与架构弹性。

**调优基准线**：以服务器 B 的存储容量和服务器 A 的 4 核 CPU 为瓶颈指标，监控到位后先扩 B 的磁盘，再扩 A 的 CPU。
